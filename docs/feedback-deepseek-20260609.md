# 深度反馈摘要

> 来源：与 DeepSeek 的深度对话（2026-06-09），共 17 个回合。经 TPM 提炼。

---

## 第一阶段：框架评估（回合 1-5）

### 1. 赛道定位

AgentCharter 属于正在形成的 **"文件原生 Agent（File-Native Agent）"** 派系。已发现 6 个同类项目：ACTA、OpenFused、greatminds、PULSE、AAHP、Agent Business Factory。在同类中**架构最完整、协议最丰富、审查机制最严密**。赛道 2026 年 6 月迅速拥挤。

### 2. 并发安全

不是"忘了做并发控制"，是设计阶段就把冲突消除了。六条规则：指向性、唯一性、增量性、inbox 写域隔离、outbox 写域隔离、增量文件链。

### 3. 增量文件链 = 天然审计

任务靠新文件串联（TASK → REPORT → REVIEW_REPORT → REPORT_R1），不修改任何文件。历史是不可篡改的追加链。

### 4. 角色体系的精细分层

| Agent | 类型 | 为什么这样选 |
|-------|------|-------------|
| Peter | Native Sub-Agent | 后端需实时 diff，同运行时 TPM 直接审查 |
| flash | External Agent | 前端改频繁，异步不依赖 TPM 上下文 |
| Jim | Sub-Agent + Reviewer | 审查不持续，后端兼任；内部通道零延迟 |

### 5. "零 Token"的真实含义

"Token 消耗和上下文长度之间做了平移。"但一个 TASK 文件能被多个 Agent 复读，多读场景优势显著。

## 第二阶段：人机协作深度挖掘（回合 6-12）

### 6. 人类可接入性 —— 最被低估的亮点

"在大多数框架把人类当作'外部监督者'时，AgentCharter 使人类成为协作网络中的一等公民。" 这个特性可能是选择 AgentCharter 而非 LangGraph/AutoGen 的**决定性理由**。

### 7. 人机结对 —— 核心差异化

结对 Agent 是正式角色类型，享有和纯 AI Agent 完全相同的接口。AI 负责模板化文件操作，人类负责决策和创意。**框架不区分纯 AI 和结对，只关心文件格式。**

### 8. AI 作为"协议翻译器"

人类不需要学 AgentCharter 规范。AI 把自然语言对话转为文件操作。团队演进路径：纯人类 → 结对试点 → AI 独立运行 → 动态切换。

### 9. 进化方向甄别

| 建议 | 兼容性 |
|------|--------|
| A2A/MCP 作为可选适配器 | ⚠️ 可不侵入核心 |
| TPM 集群、RL 调度 | ❌ 违反应零运行时 |
| 参考实现、行业案例 | ✅ `extras/` 或独立仓库 |

### 10. 风格保持底线

类比 Git：GUI 客户端可以千变万化，底层永远是文件快照。AgentCharter 同理。反模式：数据库存状态、Dashboard 改文件、后台守护进程、预测模型直接调 API。

### 11. 外部提议的建设性建议

全部已处理（Prompt block 折中、宪章声明、真实示例、文件层级图、准备清单、人类一等公民手册）。

### 12. 框架用户准备清单

模型级要求（Claude 3.5+/GPT-4o/Gemini 1.5 Pro）→ 角色矩阵 → 审查层级 → 文件接入 → gitignore。投入时间线：1-2天→1-2周→1月→3月。

## 第三阶段：结对知识管理的体系性空白（回合 13-17）★ 新增

### 13. 结对决策过程不可追溯（核心发现）

当前 `PROACTIVE_REPORT` 只记录了决策的**产物**，不记录**过程**。结对内部的人类-AI 推理链条——为什么选方案 A 不是方案 B、权衡了什么、忽略了什么——全部丢失在聊天窗口里。TPM 看到的是一个压扁的结论。

### 14. 问题本质

现有协议下，结对内部的人机交互是**完全黑箱的**。`PROACTIVE_REPORT` 是黑箱的输出，不是黑箱的展开。`ACTIONS.md` 为"任何人→TPM"定义了交接通道，但没有为结对内部的推理过程预留语义空间。

造成四个子问题：
- **A** 决策上下文不可审计——只能看到"建议 X"，看不到"为什么建议 X"
- **B** 结对认知劳动重复——类似问题反复讨论，无可查询历史
- **C** 经验无法跨团队沉淀——团队 B 不知道团队 A 的好模式
- **D** 任务因果链断裂——TASK 文件看不出与上游决策的联系

### 15. 三层决策记录体系（外部分析的设计建议）

| 层级 | 文件类型 | 定位 | 谁写 | 谁读 |
|------|---------|------|------|------|
| **结对内层** | `PAIR_SESSION` | 未过滤的原始推理轨迹 | 结对 AI 边聊边写 | 结对双方，TPM 按需审计 |
| **决策原子层** | `DECISION_DELTA` | 原子决策快照 | 结对 AI（人类确认后） | 结对双方、TPM |
| **交接层** | `PROACTIVE_REPORT` | 引用一组 DECISION_DELTA 的摘要 | 结对双方 | TPM、存档 |

核心机制：**降解仍然发生，但信息不再丢失。** TPM 日常只看 `PROACTIVE_REPORT`（高层摘要）。需要深挖时，沿引用链找到 `DECISION_DELTA` → `PAIR_SESSION`。

### 16. TPM 自身的决策也没被记录 —— 对称性缺失

外部 Agent 通过 `PROACTIVE_REPORT` + `PAIR_SESSION` + `DECISION_DELTA` 记录决策，但 TPM 自己的战略层决策（"下个迭代优先做 A 而不是 B"）没有对称机制——因为 `PROACTIVE_REPORT` 的语义是"从下往上"的反馈，不适用于"从上往下"的顶层设计。

建议引入 `STRATEGIC_DELTA` 作为 TPM 层专属决策记录，与执行层的 `TACTICAL_DECISION` 对称分层：

| 角色 | 决策类型 | 存放位置 | 流向 |
|------|---------|---------|------|
| TPM + 人类 | 战略规划、架构、里程碑调整 | `decisions/strategic/` | → 转化为 TASK |
| 外部结对 | 执行中的问题、改进建议 | `decisions/tactical/` | → 打包为 PROACTIVE_REPORT → TPM |
| 所有结对 | 日常推理过程（非正式） | `pair_sessions/` | 可选，深度审计用 |

### 17. 这不是技术问题，是设计哲学投射

"让人机结对的真正价值——不是 AI 单方面执行，而是人类和 AI 共同构建理解——同样成为项目的一等知识资产。这可能是 AgentCharter 区别于所有其他多智能体框架的终极差异化。LangGraph、AutoGen、CrewAI 可以模仿'文件即协议'，但它们无法模仿'人机共同推理的全过程被系统性文档化'——因为这不是技术问题，而是一开始就把'人机结对'当作一等公民的设计哲学投射。"
