# AgentCharter 管理计划

> **Author**: Kimi (External Agent, @kimi-pair)
> **Date**: 2026-06-09
> **Description**: 如果由 Kimi 管理 AgentCharter 项目，三阶段执行路线

---

## 一、诊断：当前最大的三个瓶颈

| 瓶颈 | 症状 | 风险等级 |
|------|------|---------|
| **双语同步税** | 每次更新需同步 CN/EN，v3.3 的 21 个 TASK 中 3 个是 EN-SYNC | 🔴 高 — 线性消耗开发者精力 |
| **零工具链** | 初始化靠 `cp -r`、升级靠 TPM 人工对比、无验证工具 | 🔴 高 — 用户 onboarding 摩擦大 |
| **单案例验证** | 仅 wolf-judge 一个深度案例，说服力有限 | 🟡 中 — 生态信任度不足 |

这三个问题不解决，任何功能扩展都是在沙滩上建楼。

---

## 二、Phase 1：减负（第 1-2 周）

**目标**：消除重复劳动，让维护成本从 O(n) 降到 O(1)。

### TASK 001：英语版本去中心化

**现状问题**：`collaboration/` 和 `collaboration_en/` 是两套完全独立的文件树，任何修改都要做两次。

**方案**：
1. 冻结 `collaboration/` 为唯一真相源（中文优先）
2. 引入自动化翻译工具（DeepL API / Claude / GPT-4）在 CI 中自动翻译
3. 每次发布时自动生成英文版，标记 `.AUTOGENERATED`
4. 英文版 issue 不接收内容修改 PR，只接收翻译错误报告

**效果**：维护者从此只改一份文件。

### TASK 002：模板验证器（最小 CLI）

**现状问题**：Agent 写错模板格式、命名不规范、字段缺失——TPM 需要人工检查。

**方案**：一个 200 行的 Python/Node 脚本，零依赖。

功能清单：
- 验证 15 种模板格式
- 验证文件名命名规范
- 验证交叉引用（TASK_NNN 是否存在对应 REPORT_NNN）
- 验证状态机合法性

**哲学兼容**：框架本身仍然是"文件即协议"，这个工具只是**可选辅助**，不强制使用。

### TASK 003：CHANGELOG 自动化

**现状问题**：CHANGELOG.md 靠人工维护，容易遗漏。

**方案**：遵循 Keep a Changelog + SemVer，用 git conventional commits 自动生成。

```bash
$ agentcharter release 3.4.0
```

自动：扫描 commits → 分类 → 更新 CHANGELOG.md → 打 git tag → 更新版本号。

## 三、Phase 2：验证（第 3-6 周）

**目标**：用 3 个不同类型的实战案例验证框架普适性，同时建立社区贡献管道。

### TASK 004：征集 3 个新实践案例

当前只有 wolf-judge（全栈桌面应用），需要补充：

| 案例类型 | 目标验证 | 理想项目 |
|---------|---------|---------|
| **纯后端/API 服务** | 无前端时的协作模式 | FastAPI / Go 微服务 |
| **AI/ML 项目** | 数据管道、模型训练任务的协作 | PyTorch / 数据处理 |
| **开源库/工具包** | 小团队、跨地域、异步协作 | npm 包 / Rust crate |

每个案例需要提交：
- `practices/{name}/README.md` — 团队设置、角色矩阵
- `practices/{name}/examples/` — 至少 3 个真实文件（TASK → REPORT → REVIEW_REPORT）
- `practices/{name}/LESSONS.md` — 与框架默认规则的差异及原因

**筛选标准**：不接受"Hello World"级别。必须有真实任务流转，有冲突、有阻塞、有决策记录。

### TASK 005：建立社区贡献规范

当前 CONTRIBUTING.md 是框架使用者的指南，缺少**项目本身的贡献指南**。

新增内容：
- 如何提交新实践案例
- 模板修改的评审流程（P0-P3 体系用在自己身上）
- 版本号升级规则
- 双语自动化翻译的反馈渠道

### TASK 006：框架自升级的闭环验证

当前 wolf-judge 已完成 v3.2→v3.3 升级，但 AgentCharter 自身还没验证过。

**行动**：当 v3.4 发布时，用 AgentCharter 自身的 TPM（Reasonix）完成 self-upgrade：
1. TPM 读取上游仓库
2. 对比自身 `collaboration/`
3. 创建 TASK 执行更新
4. 提交 REPORT
5. 记录 DECISION

这是**"吃自己的狗粮"的终极形态**。

## 四、Phase 3：扩展（第 7-12 周）

**目标**：在验证可行的基础上，解决规模化问题，但不牺牲极简哲学。

### TASK 007：分层协议（Layer 1/2/3）

基于上下文压力问题，实施**分层核心 + 按需引用**：

```
collaboration/
├── CORE.md              ← 新增（50 行，不可压缩）
├── ROLE_TPM.md          ← 由 TPM.md 重构
├── ROLE_EXTERNAL.md     ← 由 README.md + REGISTER.md 重构
├── ROLE_SUBAGENT.md     ← 新建
├── SHARED/
│   ├── CHARTER.md
│   ├── ACTIONS.md
│   ├── PROJECT.md
│   └── templates/
└── details/
    ├── lifecycle.md
    ├── review.md
    ├── archive.md
    ├── upgrade.md
    └── decision-examples.md
```

**版本策略**：
- v3.4 发布时同时提供旧版扁平结构（`collaboration/v3.3/`）和新版分层结构（`collaboration/v3.4/`）
- 让用户选择迁移时机，不强制
- wolf-judge 作为首批迁移验证

### TASK 008：模板注册表（轻量级）

当前 15 个模板全部硬编码在框架中。某些团队可能只需要 5 个，某些团队需要 20 个。

**方案**：不改核心，只提供约定。

```markdown
# PROJECT.md 新增字段

## 本项目使用的模板
| 模板 | 用途 | 来源 |
|------|------|------|
| TASK | 任务分派 | core |
| REPORT | 任务报告 | core |
| DECISION | 决策记录 | core |
| INCIDENT | 事故报告 | custom: templates/INCIDENT_*.md |
```

Agent 读取 `PROJECT.md` 的模板表，知道自己需要理解哪些模板。**不在表中的模板 = 本项目不使用**，减少认知负担。

### TASK 009：上下文记忆压缩算法

Sub-Agent 的 `context/{name}-memory.md` 有 8KB 上限。长期项目中，这个上限会被突破。

**方案**：引入时间衰减机制。

```markdown
# context/sub-agent-memory.md

## 活跃规则（永远保留）
- Git 禁令
- 前端隔离
- 技术栈: Tauri v2 + Rust + Vue 3

## 近期决策（30 天内，保留详情）
- [2026-06-10] DECISION_015: 内存管理表

## 历史决策（30 天前，仅保留标题+链接）
- [2026-05-28] DECISION_003: IPC 命名规范 → 见 archive/decisions/
```

TPM 维护 memory 文件时，自动将旧决策"压缩"为链接。Agent 需要详情时，沿链接读取 `archive/decisions/`。

### TASK 010：GitHub Action 集成（可选增强）

不是框架必需，但降低用户采纳门槛：

```yaml
name: AgentCharter Validate
on:
  pull_request:
    paths: ['collaboration/**']
jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Validate AgentCharter files
        uses: agentcharter/validate-action@v1
```

效果：PR 中修改了协作文件时，自动验证格式。不阻塞合并，只提供警告。

## 五、不做的事（同样重要）

管理项目不仅是决定做什么，更是决定**不做什么**：

| 不做的功能 | 理由 |
|-----------|------|
| ❌ Web UI / Dashboard | 违反"文件即协议"哲学，且人类已经有 `dashboard.md` |
| ❌ 实时协作 / WebSocket | 文件系统是异步的，实时性是反模式的 |
| ❌ 数据库后端 | 文件就是数据库，加数据库是倒退 |
| ❌ Agent 运行时 / SDK | 违反零运行时原则，且与 MCP/A2A 生态重叠 |
| ❌ 付费/闭源版本 | 信任哲学与商业锁天然矛盾 |
| ❌ 多语言扩展（日语/德语） | 除非有社区贡献者主动维护，否则不自研 |
| ❌ AI 代理市场 / Agent 注册中心 | 中心化服务与去中心化文件协议冲突 |

## 六、成功指标

| 阶段 | 指标 | 目标值 |
|------|------|--------|
| Phase 1 结束 | 维护一份文件 vs 两份的时间比 | 从 2x 降到 1.1x |
| Phase 1 结束 | 模板格式错误率（人工检查） | 从 ~15% 降到 <5% |
| Phase 2 结束 | 实践案例数量 | 从 1 个增至 4 个 |
| Phase 2 结束 | 外部贡献者 PR 数 | ≥3 个非作者提交 |
| Phase 3 结束 | 分层协议采纳率 | ≥50% 新项目选择 v3.4 结构 |
| Phase 3 结束 | GitHub Stars | 不能只看数字，但可作为信号 |

## 七、风险管理

| 风险 | 缓解措施 |
|------|---------|
| 自动化翻译质量差 | 在 CI 中标记 `.AUTOGENERATED`，社区可提 issue 修正 |
| 验证工具成为事实上的强制依赖 | 明确文档："框架本身零依赖，工具是可选辅助" |
| 新实践案例质量参差不齐 | 严格的案例审核：必须有真实文件、有决策记录、有 LESSONS |
| 分层协议导致规则碎片化 | Layer 1 明确声明："本文件是唯一真相源，details/ 不改变其含义" |
| 社区贡献导致哲学漂移 | 核心原则（信任而非控制、零运行时、文件即协议）写入 `CORE.md` 后冻结 |

---

## 八、一句话总结

> **先让维护者从双语同步中解放出来，再让工具验证替代人工检查，然后用三个真实案例证明普适性，最后在保持极简的前提下为规模化做准备。**

AgentCharter 的最大资产不是代码，是**"信任而非控制"的哲学共识**。管理这个项目的第一要务是：**不要让维护负担、功能膨胀或商业化冲动腐蚀这个共识。**
