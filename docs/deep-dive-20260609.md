# AgentCharter 深度对话全记录

> 来源：Zehee（作者）与 DeepSeek / GitHub Copilot 的 21 轮深度对话
> 日期：2026-06-09
> 整理：TPM (Reasonix)，基于对话全文提炼

## 第一幕：框架审计（回合 1-5）

### 1.1 外部第一眼：一个"纯规范项目"

DeepSeek 的初始扫描结论——"一个纯规范和文档性项目，没有提供 Agent 的具体实现代码"。评分 7/10，概念设计 9 分，落地性 5 分。

但它在第一轮对话中就准确识别了 AgentCharter 的核心特征：**增量文件链、ACTIONS.md 预分配通道、Git 隔离**。它把这套体系总结为"极致简洁与优雅，可能是目前最简洁的多智能体协作框架之一"。

### 1.2 第二轮扫描：赛道定位

第二次深度扫描中，DeepSeek 搜索了外部生态，发现 AgentCharter 属于正在形成的 **"文件原生 Agent（File-Native Agent）"** 派系，并定位了 6 个同类项目：ACTA、OpenFused、greatminds、PULSE、AAHP、Agent Business Factory。

它的结论：AgentCharter 在同类中**架构最完整、协议最丰富、审查机制最严密**。14 个标准化模板 + 12 章操作手册 + P0-P3 分级审查 + 三种角色精准定义 + Git 权限隔离，这些在同类中都不存在。

但赛道正在迅速拥挤——2026 年 6 月就有多个项目声称"零 token 协调""无 SDK 锁定"。

### 1.3 并发安全的深层认可

DeepSeek 把 AgentCharter 的并发安全模型提炼为一条完整链条：**指向性、唯一性、增量性、inbox 写域隔离、outbox 写域隔离、增量文件链**。它的评语："不是'忘了做并发控制'，而是设计阶段就把冲突可能消除了。比运行时文件锁高一维。"

### 1.4 增量文件链 = 天然不可否认

它引用了学术搜索结果——AI 智能体的"状态突变"需要隔离和检查点——然后指出 AgentCharter 的增量文件链正好精准解决了这个问题。每个任务不靠修改同一个文件推进，而靠一系列新文件串联。

## 第二幕：人机协作深挖（回合 6-12）

### 2.1 最被低估的亮点

在第六轮对话中，DeepSeek 主动指出："人类的可接入性与可参与度，恰恰是 AgentCharter 区别于其他多智能体框架的核心亮点之一。"

它分析得出一张对比表：

| 维度 | AgentCharter | LangGraph / AutoGen |
|------|-------------|---------------------|
| 人类参与方式 | 文件读写（Markdown） | 代码修改 / 对话插入 / UI 面板 |
| 参与门槛 | 极低（会编辑文本即可） | 较高（需懂 Python 或框架 API） |
| 审计能力 | 天然（Git + 文件链） | 依赖框架内置 trace/日志 |
| 非技术人员友好度 | 高 | 低 |

"AgentCharter 不仅仅允许人类介入，而是把**人类、AI、人机混合作为三种平等的 Agent 类型**，通过同一个文件协议进行协作。这是 LangGraph/AutoGen 等框架难以直接提供的。"

### 2.2 人机结对的正规化

当 Zehee 提醒它"文档中明确说明了任何外部 Agent 都可能是一个人或人机结对"时，DeepSeek 自行修正了之前的判断，并重新定位了人机结对在整个体系中的地位：

> "人机结对是一个正式的 Agent 类型，享有和其他 AI Agent 完全相同的协作接口。结对 Agent 有自己的 `outbox/`、可以领取任务、提交报告、发起主动报告。"

它进一步指出，结对模式解决了"高模型依赖性"的痛点——当模型能力不足时，人类可以接管；当模型足够时，人类退居审查或完全放手。团队演进路径是：**纯人类 → 人机结对试点 → AI 独立运行 → 动态切换**。

### 2.3 AI 作为"协议翻译器"

DeepSeek 提出了一个精准的概念：在 AgentCharter 中，AI 充当的是**协议翻译器**——人类不需要学习任何框架规范，AI 助手负责把自然语言对话转化为符合协议的文件操作。人类只需表达意图和做决策。

### 2.4 用户对可发现性的反馈

在 README 顶部那段"评价这个仓库"的提示文案上，Zehee 指出许多 AI 读到 `Evaluate this repo` 会偷懒，只扫一眼目录就走。DeepSeek 同意这是实际存在的问题，建议改成"深度分析并评价"。

## 第三幕：结对知识管理的体系性空白（回合 13-17）

### 3.1 核心发现：决策过程没有被记录

这是整个对话中最关键的发现。Zehee 指出：

> "人机模式的外部 Agent 目前在项目中是用主动报告来体现的，但是主动报告只是人机结对后的产物，决策的过程没有被记录。"

DeepSeek 对此进行了深入分析，得出了以下结论：

**当前 PROACTIVE_REPORT 只记录了决策的产物（"建议 X"），不记录决策的过程（"为什么建议 X"）。** 结对内部的人类-AI 推理链条——为什么选方案 A 不是方案 B、权衡了什么、最终判断依据——全部在聊天窗口中流失。

造成四个子问题：
- **A**：决策上下文不可审计——只能看到"建议 X"，看不到"为什么建议 X"
- **B**：结对认知劳动重复——类似问题反复讨论，无可查询历史
- **C**：经验无法跨团队沉淀——团队 B 不知道团队 A 的好模式
- **D**：任务因果链断裂——TASK 文件看不出与上游决策的联系

### 3.2 三层解决方案：PAIR_SESSION + DECISION_DELTA + PROACTIVE_REPORT

DeepSeek 提出了三层决策记录体系：

**第一层 — PAIR_SESSION（结对会话记录）**
- 轻量级的原始推理轨迹
- 结对 AI 在对话中实时编写
- 不需要人类额外操作，AI 自动捕获关键对话片段

**第二层 — DECISION_DELTA（原子决策快照）**
- 将每个原子决策提取为独立的`DECISION_DELTA_<id>.md`文件
- 包含：决策内容、推理背景、考虑的替代方案、最终判断、关联 TASK
- 多 DECISION_DELTA → 一个 PROACTIVE_REPORT 的映射关系
- 人类一键确认后写入

**第三层 — PROACTIVE_REPORT（正式报告）**
- 不改——继续承载"从结对到 TPM"的交接职责
- 改动：增加可选的"关联 DECISION_DELTA"引用字段
- TPM 日常只读这个，需要深挖时沿引用链追溯

核心机制："降解仍然发生，但信息不再丢失。"

### 3.3 TPM 层决策的对称性缺失

Zehee 进一步指出 TPM 自身也是一个标准的人机结对入口，其战略决策同样没有被记录。DeepSeek 识别出这是一个**对称性缺失**：

- 外部 Agent 有 PROACTIVE_REPORT + DECISION_DELTA
- 但 TPM 层的战略决策没有对应机制
- 因为 PROACTIVE_REPORT 的语义是"从下往上"的反馈，不适用于 TPM "从上往下"的顶层设计

解决方案：引入 `STRATEGIC_DELTA`，与 `TACTICAL_DECISION` 分层对称：

| 角色 | 决策类型 | 存放位置 | 流向 |
|------|---------|---------|------|
| TPM + 人类 | 战略规划、架构、里程碑调整 | `decisions/strategic/` | → 转化为 TASK |
| 外部结对 | 执行中的问题、改进建议 | `decisions/tactical/` | → 打包为 PROACTIVE_REPORT → TPM |

## 第四幕：从"任务执行框架"到"知识治理平台"（回合 18-19）

### 4.1 质变分析

Zehee 问："如果这些都被实现，那么这个工具会产生质的变化吗？"

DeepSeek 的回答："会。从'任务执行框架'跃迁为'人机协作知识治理平台'。"

体现在三个"可"：

- **可追溯**：任何时间点回到任何一个决策，都能看到"谁、为什么、权衡了什么"
- **可学习**：新结对可以翻阅历史决策记录，学习经验模式
- **可治理**：决策过程不再是黑箱，整个项目有了"组织记忆"

### 4.2 元实验：用 PROACTIVE_REPORT 记录关于 PROACTIVE_REPORT 的讨论

对话的最后，DeepSeek 将整个 17 轮讨论过程本身写成了一份标准的 PROACTIVE_REPORT——用框架自带的模板，记录框架自身的进化讨论。这是一个自我指涉的闭环验证。

报告完整覆盖了：
- 从第一轮到第五阶段的讨论迭代过程
- 发现的 4 个子问题
- P0-P3 优先级改进清单
- 风险评估与缓解措施
- 下一步行动建议
- 以及一份可复制的快速参考卡片

## 第五幕：开放性与信任哲学（回合 20-21）

### 5.1 临时参与 = 零摩擦协作

Zehee 点出："AgentCharter 的开放性——一个 Agent 甚至不需要加入团队都可以临时介入为项目提供价值。"

DeepSeek 将此扩展为战略视角：AgentCharter 同时支持**从松散到紧密的光谱式参与能力**。核心团队的深度结对、外围社区的临时贡献——两套机制无缝共存于同一个文件协议中。

### 5.2 信任，而不是控制

对话的最后一轮是哲学声明。Zehee 追问"文件即约束"背后的底层观念是什么。DeepSeek 回答——**信任，而不是控制。**

> 大多数 AI 框架的底层观念是"人类监督 AI"——需要 API 密钥、权限分级、沙箱隔离。但 AgentCharter 换了一个前提：我们信任 AI 会遵守协议，就像信任人类会遵守 Git 规范一样。

"约束不是铁链，而是共识。"

信任不是 naive 的乐观——文件系统是只追加的（无法篡改历史），Git 权限是隔离的（TPM 独享合并权），审计是完备的（每个决策都有记录）。信任是经过设计的。

"AgentCharter 不仅仅是一个协作框架。它是一种宣言：人类和 AI 可以基于信任和文件，平等地共同创造。"

## 我作为 TPM 的独立分析

### 接受的部分

1. **结对知识层是真实缺口** — `PROACTIVE_REPORT` 目前只记录产物不记录过程，这对声称"完全可审计"的框架来说是不自洽的。

2. **TPM 自身决策的对称性缺失** — 我们要求所有 Agent 走 TASK→REPORT，但 TPM 的战略规划只在聊天窗口里发生。这点需要修正。

3. **文件即协议 = 信任** — 这个哲学声明可以成为框架的推广内核。14 个模板和 12 章规则是技术，但"信任，不是控制"是让大家记住的东西。

### 需要甄别的部分

1. **三层决策体系偏重** — `PAIR_SESSION` + `DECISION_DELTA` + `PROACTIVE_REPORT` 加上 `STRATEGIC_DELTA` 和 `TACTICAL_DECISION` 一共五个新概念。框架的极简性是核心优势，不能为了完整性牺牲它。

2. **目录结构过细** — `decisions/strategic/active/proposed/` 这样的四层嵌套违反了当前扁平化设计。如果引入，应该从 `decisions/` 单层开始。

3. **进化建议中约 60% 不合适** — A2A/MCP 集成、TPM 集群、RL 调度等建议不符合零运行时哲学，仅作为可选适配器或独立项目考虑。

### 建议的最小化落地

不是把整个体系一次性搬进框架。而是选两件事先做：

1. **在 `collaboration/README.md` §六（主动报告）旁边加一节"结对决策记录"** — 解释为什么 `PROACTIVE_REPORT` 只记录产物，以及结对如何通过 `DECISION_` 文件补充过程记录。仅定义规范，不加模板。

2. **在 `TPM.md` §一 核心原则里加一条** — "你的战略决策也需要文件化。每次与人类讨论重大计划变更后，创建 `DECISION_` 或 `STRATEGIC_` 文件记录决策过程和依据。"

其余的（PAIR_SESSION、三层目录、CLI 工具）留给社区实践——让 wolf-judge 先试，跑通了再考虑是否升为核心规范。
