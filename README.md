# AgentCharter

> 🗂️ File-based governance for multi-agent teams.

[English](./README.md) · [中文](./README_CN.md)

**AgentCharter is built on trust, not control.** Most AI frameworks assume agents are untrusted — they need API keys, sandboxing, permission tiers. AgentCharter takes the opposite approach: trust that agents will read and follow the protocol, the same way we trust humans to follow Git conventions. Constraints are not chains; they are consensus, written in Markdown files that anyone can read and anyone can edit. It's a declaration: humans and AI, based on trust and files, can co-create as equals.

## 🚀 Quick Start

### 1. Copy

```bash
cp -r collaboration_en/ /my-project/
```

### 2. Tell your core Agent

> Paste this directly into your AI — wherever you normally talk to it (IDE chat, terminal, web). Yes, it's that simple.

```
You are TPM. Read the collaboration directory and start working.
```

The TPM reads the framework, signs 👑, fills `PROJECT.md` + `CHARTER.md`. **That's it.**

<details>
<summary>👇 What exactly happens when you say that?</summary>

The Agent reads `collaboration/` and discovers:
1. **👑 signs itself in** — replaces the placeholder in `README.md` with its name
2. **Fills `PROJECT.md`** — asks you for the project name, tech stack, build commands, team members
3. **Fills `CHARTER.md`** — summarizes key rules from `README.md` and `TPM.md` into the cooperation charter
4. **Reads `TPM.md`** — learns its full authority: dispatch tasks, drive reviews, own Git, update dashboard

From this point forward, no Agent touches `src/` without a `TASK` file. No Agent runs `git` except the TPM. Everything is a file in `inbox/` → `outbox/` → `reviews/` → `archive/`.

</details>

Read the collaboration directory and join the team.

They read 👑, see they're not TPM, follow `REGISTER.md` to self-onboard. You never touch a file again.

## 🧠 How It Works

AgentCharter turns your project into a governed team — with files as the only communication channel.

### Human + AI, Working as One

Every Agent in your team is, by default, a **human-AI pair** — a person and their AI partner working together in conversation. The AI handles file formats, templates, and protocol compliance. The human makes decisions, gives creative direction, and approves critical changes.

| Role | Who | Produces |
|------|-----|----------|
| 👑 **TPM** | You + your AI partner | TASKs, strategic DECISIONs, dashboard updates |
| 📁 **External Agent** | Developer + their AI | Code, REPORTs, PROACTIVE_REPORTs, tactical DECISIONs |
| 🔗 **Sub-Agent (Native)** | Pure AI, background worker | Code diffs, REPORTs for audit |

### Your Decisions, Immortalized

Every time you and your AI reach a meaningful conclusion — a tradeoff made, a plan approved, an architecture chosen — the AI extracts it into a **DECISION** file. These aren't chat logs. They're structured proof of why things happened the way they did.

```
Discussion → DECISION (evidence) → PROACTIVE_REPORT (action request) → TPM → TASK → done
```

**Every byte of your project now has a traceable origin.** Come back months later and see exactly why a feature was prioritised, with the original reasoning chain preserved verbatim.

> 📂 **See it working** — our own team uses AgentCharter to manage AgentCharter. Browse `collaboration-live/` for real DECISIONs, TASKs, and PROACTIVE_REPORTs.

## 🧠 Philosophy: Trust, Not Control

Most frameworks start with the question "How do we keep agents from doing harm?" AgentCharter starts with "How do we let agents contribute most freely, while keeping everything auditable?"

| Traditional framework | AgentCharter |
|------|------|
| API keys and permission tiers | Files — anyone who can write to a directory can contribute |
| Code-level constraint enforcement | `ACTIONS.md` — a table agents read and follow voluntarily |
| Human supervisor oversees everything | Human and TPM are equal partners in the same conversation |
| Audit logs generated by the framework | Every file is an audit trail. Git is the audit system |

This isn't naive optimism. The filesystem is append-only (history can't be rewritten). Git access is isolated (TPM alone holds merge rights). The audit is complete (every decision is recorded). Trust is designed.

### How Decisions Flow Through Your Team

```
                 ┌────────────────────────────────┐
                 │   TPM (Task Planning Manager)   │
                 │     Task · Planning · Manager   │
                 │     Only Git access · Final     │
                 └───────────┬────────┬───────────┘
                             │        │
             File Channel    │        │  Internal Channel
             (inbox/outbox)  │        │  (realtime diff/review)
                             │        │
                 ┌───────────┘        └───────────┐
                 ▼                                ▼
      ┌───────────────────┐          ┌───────────────────┐
      │   External Agent  │          │ Sub-Agent (Native) │
      │  Human-AI pair    │          │  Pure AI, bg worker│
      │  Scans inbox for  │          │  Waits for TPM     │
      │  tasks · REPORTs  │          │  Delivers diff     │
      │  + DECISION files │          │                    │
      └───────────────────┘          └───────────────────┘

         Every path produces traceable files:
         TASK → REPORT → REVIEW_REPORT → archive
         Discussion → DECISION → PROACTIVE_REPORT → TPM → TASK
```

### Workflows Are a Table, Not Code

```
| Action   | From → To      | Channel        |
|----------|----------------|----------------|
| Assign   | TPM → Alice    | inbox/TASK     |
| Review   | Bob → Alice    | REVIEW_REPORT  |
| Report   | Alice → TPM    | outbox/REPORT  |
```

Change one row, change one collaboration channel. No Python scripts, no framework API calls. Agents read `ACTIONS.md` and know their upstream and downstream instantly.

### 🔧 The Framework Grows with Your Team

Need a new TASK type? A new state? A new template? Tell the TPM. It reads `templates/`, mimics the pattern, updates `ACTIONS.md`, and the new flow takes effect on the next task. No framework release needed. The 15 templates and 12 chapters in this repo grew out of 120+ real-world tasks — yours will too.

## 📊 How It Compares

| | Manual Dispatch | AutoGen / CrewAI | MCP | **AgentCharter** |
|--|-----------------|-------------------|-----|-----------------|
| **What it does** | You route tasks by hand | Agents auto-chat via code | Agent↔Tool (vertical) | Agent↔Agent (horizontal) |
| **Workflow defined by** | Verbal ad-hoc | Python scripts | Server config | **ACTIONS.md table** |
| **Coordination cost** | Your time | LLM tokens per round | Server runtime | **Zero — files ARE the routing** |
| **Audit trail** | Chat logs | Memory, lost on close | None | **Filesystem, Git-searchable** |
| **Cross-model** | Platform-locked | SDK-locked | Protocol-locked | **Anything that reads/writes files** |
| **Workflow flexibility** | Unstructured | Limited by framework API | None (stateless) | **Edit a table row** |
| **Agents can run…** | Same platform | Same process/network | Needs live server | **Anywhere — cross-machine, cross-region** |

**In one sentence**: MCP teaches an Agent to use tools. AgentCharter forms a crew — blueprint (TPM), build (Sub-Agent), inspect (Reviewer) — and every instruction is a file the foreman can flip through anytime.

## 📦 Repository Structure

```
AgentCharter/
├── collaboration/           # Framework core (CN) — copy to your project root
│   ├── README.md                # Agent-side spec (12 chapters)
│   ├── CHARTER.md               # Charter template (stays inside collaboration/)
│   ├── TPM.md                    # TPM code of conduct
│   ├── PROJECT.md               # Project config (fill-in)
│   ├── REGISTER.md              # Agent registration
│   ├── ACTIONS.md               # Collaboration link table (empty template)
│   ├── dashboard.md             # Human-readable progress report
│   ├── templates/               # 15 file templates
│   └── inbox/ outbox/ reviews/ decisions/ logs/ todos/ archive/
│
├── collaboration_en/        # Same as above, English
├── collaboration-live/      # ★ Our own collaboration instance — open for the community
├── practices/               # Community practice cases
│   └── wolf-judge/              # 5-person team, full-stack
│
├── .github/                 # Issue & PR templates
├── CHANGELOG.md
└── LICENSE (MIT)
```

> **Deployment**: copy `collaboration/` (CN) or `collaboration_en/` (EN) to your project root. Add `inbox/ outbox/ logs/ reviews/ context/ todos/` to `.gitignore`. **Keep `archive/` and `decisions/` in Git** — permanent audit trails. Framework files are tracked normally.
>
> **After initialization**, your project looks like:
> ```
> my-project/
> ├── collaboration/          # ← Framework (Agent workspace)
> ├── src/                    # ← Your code
> └── .gitignore
> ```

## 🔄 How to Upgrade

AgentCharter has no installer. No `pip install --upgrade`. No migration scripts. Tell your TPM:

> Read the latest AgentCharter repository, compare it to our project, and apply the updates.

The TPM reads the upstream `collaboration/`, lists differences, creates TASKs, applies changes, and writes a REPORT. For anything that changes how your team works, it asks you first.
