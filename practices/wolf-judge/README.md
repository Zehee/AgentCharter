# 实践：新狼官（wolf-judge-assistant）

> 5 人团队 · Tauri v2 + Rust + Vue 3 · 桌面应用 · 120+ 任务闭环
>
> 本案例展示了一个**前后端分离、多 Agent 混合类型**的协作范式。

---

## 一、项目背景

**新狼官**是一个狼人杀桌面法官助手，帮助面杀局管理回合、角色、投票、复盘。技术栈 Tauri v2（桌面壳）+ Rust（后端状态机）+ Vue 3（前端 UI）。

选择 AgentCharter 的原因：项目有明确的前后端分工、多种 Agent 运行环境，需要一个框架让多个 AI 在同一代码库中协同而不互相覆盖。

## 二、角色设置与职责矩阵

### 2.1 团队全景

```

                           │         Kimi (TPM)            │
                           │  Native Host · 唯一 Git 权限  │
                           │  决策 · 分派 · 审批 · 归档    │
                           └────────┬──────────┬──────────┘
                                    │          │
                   ┌────────────────┼──────────┼──────────────────┐
                   │                │          │                  │
                   ▼                ▼          ▼                  ▼
         ┌─────────────────┐ ┌──────────┐ ┌──────────┐ ┌─────────────────┐
         │ flash (External) │ │  Peter   │ │   Jim    │ │Designer(External)│
         │  前端 Vue 3 / TS │ │ (Native) │ │ (Native) │ │  视觉审查·设计提案 │
         │  专项审查·测试    │ │ 后端 Rust│ │ 后端 Rust│ │   兼任 Reporter  │
         └───────┬─────────┘ └────┬─────┘ │CodeReview │ └───────┬─────────┘

                 │               │            │               │
                 ▼               ▼            ▼               ▼
         ┌────────────────────────────────────────────────────────────┐
         │                     buddy (External)                       │
         │               手动测试 · E2E 验证                          │
         └────────────────────────────────────────────────────────────┘
```

### 2.2 角色职责表

| Agent | 类型 | 运行时 | 核心职责 | 兼任角色 |
|-------|------|--------|----------|----------|
| **Kimi** | TPM (Native Host) | 当前 IDE | 任务分派、架构决策、Git 操作、最终审批、归档 | Sub-Agent 宿主 |
| **flash** | External Agent | 独立环境 | Vue 3 组件开发、TypeScript、UI/UX、前端测试、跨端专项审查 | — |
| **Peter** | Native Sub-Agent | 同 IDE 后台 | Rust 后端开发、状态机、SQLite、IPC 命令实现 | — |
| **Jim** | Native Sub-Agent | 同 IDE 后台 | Rust 后端开发、数据库迁移 | **CodeReviewer**（审查所有代码） |
| **Designer** | External Agent | 独立环境 | 前端视觉审查、交互分析、设计提案、UI 规范 | **Reporter**（提交主动报告） |
| **buddy** | External Agent | 独立环境 | 手动测试、回归验证、E2E 测试执行 | — |

### 2.3 为什么这样设置？

**Peter 为什么是 Native Sub-Agent？**
- 后端代码变更需要实时 diff 交付给 Kimi（内部通道更高效）
- 同运行时，Kimi 可以直接审查 diff 质量，减少文件通道往返

**flash 为什么是 External Agent？**
- 前端开发稳定，可并行处理多个任务
- External Agent 不依赖 Kimi 上下文，可独立长时间运行
- 前端改动频繁，文件通道的异步模式更适合

**Jim 为什么一人两角？**
- 代码审查不需要持续工作，充分利用闲置
- 审查需要深度理解后端代码，让后端开发者兼任最合理
- 审查结论通过内部通道通知 Kimi，零延迟

**Designer 为什么是 Reporter？**
- 设计师的工作是"发现 → 提议"，不是"领取任务 → 编码"
- 主动报告（阅后即焚）模式天然匹配设计审查场景
- 不做标准任务流转，不占 inbox 队列

### 3.1 任务分派流

```
用户需求 / Kimi 决策
        │
        ▼
    Kimi 判断：这是谁的活？
        │
        ├── 前端 → 写 inbox/TASK_NNN_xxx_FLASH.md → flash 巡检领取
```

### 3.3 完整协作链路表

| 动作 | 发起方 → 接收方 | 通道 | 说明 |
|------|----------------|------|------|
| 分配任务 | Kimi → flash | `inbox/TASK` | flash 自主巡检 |
| 分配任务 | Kimi → Peter | 内部通道 + `inbox/TASK`(记录) | Peter 等待注入 |
| 分配任务 | Kimi → buddy | `inbox/TASK_TEST` | buddy 自主巡检 |
| 指派设计 | 用户 → Designer | 直接交付 | 不经过 inbox |
| 提交报告 | flash → Kimi | `outbox/REPORT` | 文件通道 |
| 提交报告 | Peter → Kimi | 内部通道(diff) + `outbox/REPORT` | 审计用 |
| 提交设计报告 | Designer → Kimi | `outbox/PROACTIVE_REPORT` | 阅后即焚 |
| 提交测试报告 | buddy → Kimi | `outbox/TEST_REPORT` | 文件通道 |
| 唤醒审查 | Kimi → Jim | 内部通道（只发 REPORT 编号） | 轻量通知 |
| 审查代码 | Jim → flash/Peter | `REPORT → REVIEW_REPORT` | Jim 直接闭环 |
| 审查结论 | Jim → flash/Peter | `reviews/REVIEW_REPORT` | 执行者自读 |
| 质量确认 | Jim → Kimi | 内部通道 "ACCEPT 通知" | 标准化格式 |
| P0 通过 | — | Kimi 直接 commit | 无 Jim 参与 |
| 阻塞通知 | 任意 → 任意 | `outbox/BLOCKING` | 被阻塞方读 |
| 阻塞解除 | 解除方 → 阻塞方 | `outbox/BLOCKING_REPLY` | — |

## 四、个性化规则详解

以下规则是**在本项目实例中演化出来的**，并非 AgentCharter 框架强制要求。

### 4.1 纪律红线

这些是针对具体角色和代码库结构设立的硬性约束：

| 规则 | 内容 | 适用范围 | 产生背景 |
|------|------|----------|----------|
| **Git 禁令** | 任何 agent 严禁执行任何 git 命令（status、log、commit、diff、checkout 等），一刀切，无白名单 | 所有 agent | 防止 agent 自动提交不完整代码 |
| **前端隔离** | Peter 严禁修改任何前端文件（`api/index.ts`、`views/`、`components/`、`stores/` 等） | Peter | 历史多次违规修改前端文件 |
| **越界报告** | 发现工作目录问题、权限问题时立即报告 Kimi，不得自行处理 | Peter、Jim | Sub-Agent 无全局判断能力 |

### 4.2 审查分工矩阵

| 产出者 | 代码审查者 | 设计审查者 | 最终决策 |
|--------|-----------|-----------|----------|
| Peter（后端） | Jim (CodeReviewer) | — | Kimi |
| flash（前端） | Jim (CodeReviewer) | Designer | Kimi |
| Designer（设计） | — | Kimi | Kimi |

**跨端改动特殊流程**（新增/修改 IPC 命令、数据结构变更、共享模型修改）：
```
Jim 代码审查 → flash 专项审查（前端视角验证）→ Kimi 最终决策
```

### 4.3 任务分发策略

> 核心目标：不让 agents 空闲，但避免 Sub-Agent 超时。

| Agent | 策略 | 并发上限 | 说明 |
|-------|------|----------|------|
| **Peter** (Native Sub-Agent) | 尽量少 | 1-2 个 | Native Sub-Agent 上下文不足易超时 |
| **flash** (External) | 可以多一点 | 不限 | 独立环境，稳定运行 |
| **Jim** (Native Sub-Agent) | 按需唤醒 | 不通过 inbox 分派 | Kimi 在代码产出后主动唤醒 |
| **Designer** (External) | 主动指派 | 不限 | 用户指派 → 完成 → 提交主动报告 |

**分发原则**：
1. inbox 为空时立即分配新任务，保持流动
2. 每个成员尽量持有 ≥2 个任务（Peter 除外）
3. 已确认的需求全部写成 TASK 放队列，避免遗忘
4. 复杂任务拆细到 1-2 天可交付

### 4.4 P0-P3 分级审查

> 这是本实践对框架审查机制的**实例化定制**。

| 级别 | 判定标准 | 审查深度 | Kimi 消耗 |
|------|----------|----------|-----------|
| **P0** | 单文件、纯 UI/样式/文案/格式化 | 无审查，Kimi 直接 commit | 0 |
| **P1** | 2-3 文件、组件级逻辑 | Jim 审 → 摘要 | 5-10 行 |
| **P2** | 跨模块、数据流、新增 IPC | Jim 审 → 完整报告 → Kimi 确认 | 摘要+意见 |
| **P3** | 架构/安全/核心流程 | Jim 审 → Kimi 深度验证 | 完整介入 |

**P0 白名单**（严格限定，宁高勿低）：CSS 调整、文案修改、图标替换、布局微调、配置非逻辑修改、格式化

### 4.5 归档规则（精细版）

在框架默认规则之上，实例演化出**已读确认制**：

```
流程中间文件（TASK/REVISION/BLOCKING/TODO）
  → 处理完即归档，价值已归零

流程末端文件（REPORT/REVIEW_REPORT/NOTICE/REPLY）
  → 外部 agent 在文件顶部标注 "✅ 已读 BY {AGENT} @ {DATE}"
  → Kimi 确认所有相关方已读后归档
  → 例外：仅到 Kimi 的文件处理完即可归档
```

### 4.6 Sub-Agent 上下文记忆

Native Sub-Agent (Peter/Jim) 在 IDE 重启后丢失上下文。解决方式：

- 每个 Sub-Agent 有 `context/{name}-memory.md`，由 Kimi 维护
- 内容包含：项目技术栈基准、历史陷阱清单、纪律红线、审查标准、活跃任务状态
- Kimi 每次唤起 Sub-Agent 时，将 memory 文件**完整注入 prompt 顶部**
- 存储上限 8KB，超限时将旧决策归档到 `*-memory-archive.md`

### 4.7 审查摘要流转

解决多轮审查时 Reviewer 上下文不稳定的机制：

```
R0: Jim 写 REVIEW_REPORT（【审查摘要】= R0 评分+问题）
  ↓
R1: 执行者写 REPORT_R1（复制 R0 原文，追加 R1 修复回应）
  ↓ Jim 读 REPORT_R1，无需翻上轮文件
R1: Jim 写 REVIEW_REPORT_R1（复制 R0 + 追加 R1 结论）
  ↓
... 循环直到 ACCEPT
  ↓ Kimi 只读最后一轮摘要，3+ 轮自动触发关注
```

### 4.8 开发模式

| 维度 | 设置 |
|------|------|
| 迭代周期 | 无固定 Sprint，看板驱动，任务完成即交付 |
| 设计基线 | 变更走轻量评审（Kimi 快速批准） |
| 交付节奏 | 持续流动，完成即审查即合并 |
| 变更控制 | 取消重流程，需求变更由 Kimi 评估后直接分派 |

## 六、为什么值得参考

1. **混合 Agent 类型示范** — 同一项目中同时使用 External Agent (flash/Designer/buddy) 和 Native Sub-Agent (Peter/Jim)，展示了两种 Agent 的协作差异
2. **CodeReviewer 兼任模式** — Jim 同时写后端代码和审查他人代码，证明了"一人两角"可行
3. **分级审查体系** — P0-P3 四级审查是本实践的核心创新，大幅节省 TPM 决策成本
4. **主动报告用于设计审查** — Designer 不走标准任务流，用 Proactive Report 做"阅后即焚"式交互
5. **纪律红线** — 针对具体代码库设定的硬性约束，展示框架如何适配项目特有风险

## 七、经验教训

### ✅ 验证有效的模式

- **审查直接闭环** — Jim 直接面向执行者，不经过 Kimi 中转，节省大量上下文
- **Native Sub-Agent 后端开发** — 内部通道 diff 实时交付，审计用文件仅存记录
- **上下文记忆文件** — Native Sub-Agent 真正的痛点解决方案

### ⚠️ 踩过的坑

- **Sub-Agent 超时** — Native 实例上下文不足时容易超时，控制 1-2 个活跃任务
- **前后端 IPC 不同步** — 跨端改动审查容易遗漏，追加 flash 专项审查
- **归档时机混乱** — 早期没有规则，后来确立"中间文件即归档，末端文件待已读"
- **Agent 越界修改** — Peter 曾修改前端文件，后来设定硬性禁令
